step7

Flutterアプリ「現場ノート」に、ホーム画面ウィジェットからのクイック記録機能を追加してください。

目的:
- アプリを開かずに1タップで作業記録を仮保存できるようにする
- MVP要件の「ウィジェット対応（仮保存のみ）」を実装する

前提:
- 既存の作業履歴テーブルがある
- 作業履歴は SQLite に保存している
- ステータスに「未整理」がある
- 通常アプリ内でもクイック記録機能が存在する、または同等の処理を作れるようにする

実装方針:
- まずは Android のホーム画面ウィジェットを実装対象とする
- iOS ウィジェットは今回は対象外でよい
- ウィジェットからできる操作は「クイック記録を1件追加」のみ
- 追加されるデータは仮保存レコードとする
- 詳細入力（物件、請求先、写真、メモ編集など）はアプリ本体で行う
- GPS 取得に失敗しても記録自体は保存する
- オフラインでも保存できること
- 可能な限り既存の記録作成ロジックを再利用すること

追加仕様:
1. ウィジェット仕様
- ホーム画面に 1x1 もしくは最小サイズ相当のシンプルなウィジェットを追加
- 表示内容:
  - アプリアイコンまたは簡易ラベル
  - 「記録」ボタン相当の見た目
- タップ時:
  - 直接クイック記録を保存する
  - 保存成功後、短い Toast などで「記録しました」を表示してよい
  - 必要ならアプリを開かずバックグラウンド処理で完結してよい
- 代替として、実装都合上は「タップ → 非表示の処理 → 必要最小限の起動」でもよいが、UXはできるだけ1アクションに近づけること

2. 保存するデータ
新規作業履歴レコードとして以下を保存:
- id: UUID
- 日時: 保存時の現在時刻
- 緯度: 取得できた場合のみ保存
- 経度: 取得できた場合のみ保存
- 物件ID: null
- 請求先ID: null
- メモ: 空文字または null
- ステータス: 未整理
- 作業種別: 未設定または null（既存設計に合わせる）
- created_at / updated_at があれば適切に設定

3. 位置情報の扱い
- ウィジェット起動時に現在地取得を試みる
- 取得失敗時は緯度経度 null で保存
- 権限未許可でもアプリが落ちないこと
- 権限がない場合でも最低限「日時付き未整理レコード」を残せること
- 位置情報取得の責務はサービス層に寄せ、UI層やウィジェット実装にロジックを分散させすぎないこと

4. アプリ起動後の導線
- 保存された仮保存レコードは一覧の先頭付近に表示されること
- 未整理一覧でも見つけられること
- 必要なら詳細画面へ遷移して後から編集できること

5. アーキテクチャ
- Widget から直接 DB 操作をベタ書きしない
- 可能なら以下のように責務分離すること:
  - quick_record_service
  - work_record_repository
  - location_service
- 通常アプリのクイック記録と Widget のクイック記録が同じユースケースを通る設計が望ましい

6. Android実装
- Android ホーム画面ウィジェットを追加
- 必要な receiver / provider / manifest 設定を行う
- Flutter 側との連携には platform channel 等の適切な方法を使う
- Widgetタップ時に Flutter 側の共通記録処理を呼ぶ、または Android 側で必要最小限の保存処理を実装する
- ただし将来的な保守性を考え、記録仕様が二重管理になりにくい構成にすること

7. エラーハンドリング
- 保存失敗時にクラッシュしない
- 例外は握りつぶさずログ出力する
- 可能ならユーザーへ簡易メッセージを出す
- 権限エラー、DBエラー、位置情報取得失敗を分けて扱える構造が望ましい

8. 実装後に確認したいこと
- ウィジェットをホーム画面へ追加できる
- ウィジェットをタップすると作業履歴が1件追加される
- ステータスが未整理で保存される
- 位置情報が取れれば保存される
- 位置情報が取れなくても保存される
- アプリ再起動後も SQLite に残っている
- 一覧画面から詳細編集へ進める

9. 今回はやらないこと
- 複数ボタン付き高機能ウィジェット
- ウィジェット上で物件選択
- ウィジェット上で写真追加
- iOSウィジェット対応
- バックグラウンド常時GPS
- 通知連携

10. 完了条件
- Androidでホーム画面ウィジェットが追加できる
- ウィジェットタップで未整理レコードがSQLiteに保存される
- 位置情報が取れない環境でも最低限の記録保存ができる
- 既存一覧画面で保存結果を確認できる
- 必要なファイル追加・設定変更・実装説明を行う
- README またはコメントでセットアップ手順が分かるようにする

補足:
- MVPでは「とにかく記録を取り逃さない」ことを優先する
- 見た目よりも、保存成功率・壊れにくさ・保守しやすさを優先する
- 既存コード規約に合わせて命名・構成を整えること
